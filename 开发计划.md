# QuickNote V2 开发计划

## 1. 开发目标
- 按“可运行、可验证、可迭代”的节奏完成 QuickNote V2。
- 每一阶段都保证 `python main.py` 可运行。
- 每完成一个功能，同步更新 `开发功能点清单.md`（状态 + 简要实现说明）。

## 2. 迭代节奏（建议 6 个阶段）

## 阶段 0：工程基线（已完成）
- 目标：建立文档、依赖、运行入口、Git 仓库管理。
- 产出：
  - 需求/功能/架构/数据库文档。
  - `requirements.txt`、`main.py` 占位入口。
  - 远程仓库初始化与首个提交。
- 验收：`python main.py` 可运行；`git status` 干净。

## 阶段 1：最小可用 UI + 数据层
- 目标：可创建、查看、编辑、删除便签（文本版）。
- 实现范围：
  - `src/` 基础目录落地。
  - PySide6 主窗口、列表区、编辑区。
  - SQLite 连接与 `notes` CRUD。
  - 自动保存（节流写入）。
- 验收：
  - 新建便签后可持久化。
  - 重启应用可恢复便签内容。

## 阶段 2：标签、颜色、检索
- 目标：便签组织能力可用。
- 实现范围：
  - `tags`、`note_tags` 数据与 UI。
  - 颜色切换、Emoji 图标。
  - 关键词搜索 + 标签过滤。
- 验收：
  - 标签筛选结果正确。
  - 搜索匹配标题/正文可用。

## 阶段 3：窗口行为与低打扰交互
- 目标：悬浮便签体验成型。
- 实现范围：
  - 置顶 Pin、透明度调节。
  - 右侧贴边与悬停展开。
  - 收起模式（仅列表）。
- 验收：
  - 贴边与展开稳定，不误触抢焦点。
  - 透明度与置顶状态可保存。

## 阶段 4：提醒与通知
- 目标：任务闭环可用。
- 实现范围：
  - 提醒时间设置、重复规则（先支持单次 + 每日）。
  - APScheduler 调度与本地通知。
  - 稍后提醒（Snooze）。
- 验收：
  - 到点通知触发准确。
  - 点击通知可定位便签。

## 阶段 5：图片混排 + 语音转写 + 数据导入导出
- 目标：补齐高价值增强功能。
- 实现范围：
  - 粘贴图片、正文混排显示。
  - 语音转写（硅基流动 API）接入与错误处理。
  - Markdown/资源包导入导出。
- 验收：
  - 图片与文本混排稳定。
  - 语音转写结果可插入便签。
  - 导出后可在新环境导入恢复。

## 3. 分支与提交策略
- 分支策略：
  - `main`：稳定主线。
  - `feature/<模块>-<功能>`：功能开发。
  - `fix/<模块>-<问题>`：问题修复。
- 提交粒度：
  - 每个可验证功能点至少一次提交。
  - 提交信息遵循：`feat/fix/docs/refactor/chore`。

## 4. 质量门禁（每阶段都执行）
- 运行检查：`python main.py`
- 基础自测：
  - 新建/编辑/删除
  - 重启持久化
  - 对应阶段新增功能回归
- 文档同步：
  - 更新 `开发功能点清单.md`
  - 必要时更新 `README.md`

## 5. 当前执行顺序（从现在开始）
1. 落地 `src/` 项目骨架与基础窗口（阶段1）。
2. 完成 notes 表与 CRUD + 自动保存（阶段1）。
3. 完成标签、颜色、搜索（阶段2）。
4. 再进入窗口行为优化与提醒（阶段3~4）。
